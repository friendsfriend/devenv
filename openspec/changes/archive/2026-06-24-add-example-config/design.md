## Context

DevEnv already reads split JSON definitions, compose files, Dockerfiles, libraries, infrastructure services, and scripts from convention-based locations. First-time users must currently assemble those files manually before they can see a meaningful TUI.

The new command should stay outside the service container because it only writes static files. It should reuse the same path conventions as runtime code:

- Config directory: `DEVENV_CONFIG_DIR` or `~/.config/devenv`.
- Home directory: `DEVENV_HOME` or `~/devenv` for initial generation.
- Scripts directory: `$DEVENV_HOME/scripts`.

## Goals / Non-Goals

**Goals:**

- Create one safe command that writes a complete runnable example setup.
- Demonstrate apps, libraries, shared infrastructure, per-app infrastructure, compose profiles, Docker build/test workflows, and script runtimes.
- Use public GitHub repositories so clone/build/test flows are real.
- Fail before writing anything when existing user config or scripts are present.
- Keep implementation dependency-free and file-template based.

**Non-Goals:**

- Do not generate source repositories locally.
- Do not start Docker containers or clone repositories during config generation.
- Do not validate third-party repositories beyond providing usable config and Dockerfiles.
- Do not add TUI screens or server endpoints.
- Do not overwrite, merge, or migrate existing user configuration.

## Decisions

### Add a Cobra command, not a server API

Add `devenv create-example-config` under `server/cmd`. The command is one-shot file generation and does not need Docker, state DB, provider store, or HTTP server startup.

Alternative considered: expose an API/TUI action. Rejected because the safety contract is easier as a CLI command and avoids booting services before config exists.

### Generate files from embedded literals

Use a small generator package with static string templates and a list of output files. Before writing, scan all guarded locations. If any guarded location exists and is non-empty, return an error and write nothing.

Alternative considered: store template files in a separate directory. Rejected for now because embedding plain strings keeps packaging simple for the current single-binary build.

### Guard both config and scripts

The command fails if the resolved config directory is non-empty or if `$DEVENV_HOME/scripts` is non-empty. Missing directories are allowed. Empty directories are allowed.

This matches the first-time onboarding use case and prevents accidental pollution of user-owned config or script collections.

### Use public repos but own DevEnv Dockerfiles

Generated app/library definitions point at public GitHub repositories:

- Go REST API app: `https://github.com/pauljamescleary/go-rest-postgres.git`
- Bun/TypeScript app: `https://github.com/stevedylandev/bhvr-site.git`
- Bun TypeScript library: `https://github.com/wobsoriano/bun-lib-starter.git`

DevEnv build/test Dockerfiles live in config and are generated by the command. This shows DevEnv's external build/test convention without requiring upstream repos to carry DevEnv-specific files.

### Demonstrate shared and per-app infrastructure separately

Shared infrastructure appears in the Infrastructure tab:

- `postgres`
- `redis`
- `mailpit`

The Go app demonstrates shared infrastructure by joining the shared example network and connecting to the shared Postgres service. The Bun/TypeScript app demonstrates per-app infrastructure with profile-specific compose files that include app-local services.

### Compose profiles are filename profiles

Profiles use the existing filename convention:

- `bhvr-site-compose.yml` for default startup.
- `bhvr-site-debug-compose.yml` for debug startup.
- `bhvr-site-with-redis-compose.yml` for startup with per-app Redis.

No schema changes are needed.

## Risks / Trade-offs

- Public repos can change or disappear → use simple, established repos and keep generated Dockerfiles small enough to adjust later.
- Cross-compose shared infrastructure depends on startup order → document via generated scripts and compose names; users start infra first, then the app.
- Strict non-empty checks can block users with harmless files → this is intentional for a first-time command; users can move files or choose a clean `DEVENV_CONFIG_DIR`/`DEVENV_HOME`.
- Generated examples may need Docker network coordination → include the same named network in infra and app compose files.
